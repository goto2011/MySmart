01-Git&Repo&工具链
  • Git
    • Git原理
      • 仓库就是长江黄河，长期存在，变化永无休止；
      • 分支就是长江黄河上的各个分支，分分合合；
      • 标签就是河流上的各个堤坝，一旦建成，就固定不动。基于标签下代码，基于标签出版本，基于标签拉分支。
      • Git库: 就是根目录下的.git 目录，其中存放的是我们所提交的文档索引内容，Git 可基于文档索引内容对其所管理的文档进行内容追踪，从而实现文档的版本控制.
      • Git库中由三部分组成
        • 工作目录：用户本地的目录
        • Index（索引）：将工作目录下所有文件（包含子目录）生成快照，存放到一个临时的存储区域.
        • 仓库：将索引通过commit命令提交至仓库中，每一次提交都意味着版本在进行一次更新。
      • 每次提交都会产生一个新的changeid, 并可在该changeid上打标签, 拉分支.
      • 几个特殊的tagid
        • dev表示发生在dev分支上的最后一次提交。
        • dev^表示发生在dev分支上的倒数第二次提交。
        • dev~2表示发生在dev分支上的倒数第三次提交。
    • 查看提交记录: git log
      • -p: 显示最多信息, 包括代码变更
      • 日期范围： --after, --before
        • --before '10-01-2019'
      • 指定文件： --pathname
      • --stat -summary: 查看详细的统计数据
      • --oneline: 一行显示
      • -数字: 显示多少条
      • --pretty=raw: 显示更多提示信息
      • --graph: 显示提交线索, 主要用于父子提交
      • --author ganduan: 指定提交人
      • --grep ***: 指定关键字
      • --date_order: 按照提交时间顺序. 不准, 推荐使用 greeit 网站.
    • Git命令
      • git clone <仓名>
      • git checkout <分支名>
      • 刷新代码: git pull
      • 修改上一次提交的msg: git commit --amend
      • 推代码到服务器： git push
        • 强制push： --force
      • 获取服务器分支名: git branch -r
      • 获取本地分支名: git branch, 在列出的分支中，若分支名前有*，即表示为当前分支。
      • 查看当前代码是从哪个分支上拉的： git branch -vv
      • 获取服务器仓名： git remote -v
      • 查看状态: git status
        • git add 前后建议各 git status 一次, 确认没有状态错误.
      • 添加文件: git add
        • git add . : 添加当前目录下的所有文件和子目录  
        • git add dirname: 添加一个目录
        • git add *.java: 添加当前目录下的所有java文件
        • git add -A: 当增, 改, 删同时存在时, 用这个命令.
      • 提交: git commit
        • 直接调用git commit命令，会提示填写注释。无注释会提交失败.
        • 带注释提交是这样的: git commit -m "Initial commit of gittutor reposistory" 
        • 每一次提交，git就会为全局代码建立一个唯一的commit标识代码.
      • 追加提交: git commit --amend
        • 如果不想chang-id改变，需要在注释的最后一行注明：
        • chang-id:
      • 回退所有本地修改: git reset --hard <tagid>
        • git reset HEAD^ 回退第一个记录
        • git reset HEAD~2 回退第二个记录  
      • 回退指定文件的修改： git checkout <tagid> <filepath>
      • 查看当前仓库地址（含仓库名）： git remote show origin
      • rebase: 对提交记录进行增删改
        • 仅针对本地分支
        • 合并最后三次提交
          • git rebase -i HEAD~3
          • 在弹出窗口中编辑指令
            • s: 和前一笔合并，且保留注释；
            • f: 和前一笔合并，且不保留注释。
            • d: 删除本笔提交。
      • 删除指定提交
        • 找到它之前的提交, 执行命令: git rebase -i ID
        • 在弹出界面上, 将要删除的提交的id 前面的 pick 改为 drop, 然后保存退出.
      • 查看分支是从哪个分支拉取的(拉分支): git reflog --date= local | grep 分支名
    • Git其它命令
      • 获取帮助: git help
      • 配置用户名: git config --global user.name "userName"  
      • 配置email: git config --global user.email you@yourdomain
      • 初始化本地库: git init
        • 此时, 系统会自动创建一个名为“master”的分支。
      • 创建本地分支 work. --强烈建议。
        • git branch work
        • git checkout work
        • 或者合并成一步： git branch -b work
      • 查看当前在哪个分支上: cat .git/HEAD. --返回本地分支，没用。
      • 切换到指定分支: git checkout 分支名
      • 删除分支: git branch -D 分支名
      • 比较两个分支上的文件的区别: git diff master 分支名（比较主分支和另一个分支的区别）
      • 查看当前分支的操作记录: git whatchanged
      • 删除文件: git rm
        • git rm abc.java : 删除当前目录的指定文件
        • git rm -r *: 删除当前目录下的所有文件和子目录
        • git rm --ached adb.java: 删除指定文件，但只删除index中的文件记录；将已经git add的文件remove到cache中,这样commit的时候不会提交这个文件, 适用于一下子添加了很多文件, 却又想排除其中个别几个文件的情况.  
      • 查看指定版本的修改记录: git show + changeid
        • 查看指定标签的修改记录: git show tagName
      • 打标签: git tag "...", 可以用任意字符串.
      • 把服务器上下载下来的代码和本地代码合并, 或本地分支合并: git merge dev
      • 查看当前的git库中有哪些文件: git-ls-files
      • 本地代码更新到服务器端代码库上: git push origin , origin 是服务器端代码库名.
    • 文件忽略
      • 工作目录中有一些文件是不希望接受Git 管理的，譬如程序编译时生成的中间文件等等。Git 提供了文档忽略机制，可以将不希望接受Git 管理的文件信息写到同一目录下的.gitignore 文件中。
      • 支持 *.lib 通配符, build/ 文件路径 这两种形式.
  • cherry-pick，简称cp，同步提交
    • 将一个分支的部分提交, 移动到同主干的其它分支, 叫 cherry-pick.
    • git cherry-pick <changeid>: 将指定changeid的提交, 合并到当前分支上.
    • git cherry-pick <changeid 1> <changeid 2>: 合并指定的两个提交.
    • git cherry-pick <changeid 1>..<changeid 2>: 合并从1到2之间的所有提交. 注意, changeid 1不在内.
    • 关键参数
      • -x: 在提交信息的末尾追加一行(cherry picked from commit ...)，方便以后查到这个提交是如何产生的。
      • -s: 在提交信息的末尾追加一行操作者的签名，表示是谁进行了这个操作。
    • 如果成功，直接 git push 即可。
    • 如果发生代码冲突，Cherry pick 会停下来，让用户决定如何继续。
      • 用户先解决代码冲突。查找"===" 可找到冲突点。
      • 然后，git add .
      • 接下来，git cherry-pick --continue，继续执行cp。
      • 最后， git push ...
    • 代码冲突处的标识
      • <<<<<<< HEAD
      • 主干代码
      • =======
      • 分支代码
      • >>>>>>> 分支名
    • 代码冲突时，如果要放弃cp，则:
      • --abort ：放弃合并，回到操作前的样子。
      • --quit ： 退出 Cherry pick，但是不回到操作前的样子。

  • diff文件的生成和使用
    • git status // diff前先查看下是个好习惯
    • git diff --full-index *** > diff.txt, 其中 *** 包括如下几种:
      • 什么也不填, 即表示所有.
      • *.java, 所有的java 文件
      • a.java, 指定文件名

  • Gerrit检索
    • Gerrit是业界主流的基于git的web端的代码管理平台。检索语法如下：
    • 直接搜文件名，可获取涉及到的提交列表。
    • project: 指定仓名
    • branch: 指定分支
    • message: 指定提交备注
    • status: 指定提交状态，状态有 merged, open
    • age: 1d, 表示1天内的提交，1w表示一周的。
    • change: 指定change ID
    • commit: 指定commit ID
    • owner: 指定修改人
      • owner:self, 搜自己
    • ownerin: 指定修改人群组
    • reviewer: 指定检视人
    • reviewerin: 指定检视人群组
    • topic: 指定提交标题(一般没有)
    • 多个条件用空格即可。

  • git 常见异常
    • 问题: branch is published, but not merged
      • 创建新的本地分支： repo start work2
    • 问题: failed to push some refs to
      • 一般原因是代码冲突。
      • 直接 git rebase, 将本地分支更新到本地。
    • 问题: repo start 本地分支后，本地修改的代码丢失
      • git reflog: 查看被删除的提交id
      • git cheery-pick <提交id>, 即可合并到当前代码。
    • 问题: push后发现有代码冲突
      • 常见的情况，因为远端新的提交防不胜防。
      • git pull --rebase
      • git status: 查看出现冲突的是哪个文件
      • 打开对应文件，搜 << 定位冲突点。上面是服务器代码，下面是自己修改的代码。解即可。
      • git rebase --continue
      • 重新走提交流程。
    • 问题: git commit --amend 追加错了提交
      • 这是较常见的问题，因为上一次提交往往非如所愿。
      • git reflog, 查看操作记录
      • git reset --soft HEAD@{1} , 回退到第一步操作
      • git status
      • 会发现状态回到 commit之前。
    • 问题： checkout一个分支时报错： did not match any file(s) known to git
      • 排除分支名错误后，先git pull, 从服务器同步所有分支信息，然后再checkout。即可。

      • linux kerenl：GPL；
      • 各种运行时函数库，比如bionic：BSD；
      • Dalvik虚拟机及java类库：Apache。
        • 社区对android选择ASL的分析:
        • ASL和BSD鼓励开源软件的商业性使用，使软件公司愿意对开源软件投资，因为这些公司可以从中获利。
        • 就手机厂商而言，他们才是受android许可证影响最大的开发者，因为他们是手机平台主要的发布渠道。
        • apache允许手机厂商对平台进行改造，使得android变得他们的独家平台一样。
    • Android是怎么完成开源区隔的？
      • Android的做法是，通过bionic libc替代glibc，确保“keep GPL out of user_space”，这句话来自2008年google IO大会一篇著名ppt的原文。
      • 但是，这种做法引起了一些疑问:
        • 使用glibc. 使用系统调用，本来就没有开源义务。所以这个理由基本不成立。
        • bionic libc拷贝内核头文件，并去掉其中的GPL申明。
          • 这一点引起的争议很大。
          • google的辩解是这些是代码自动抓取的，以保证数据结构的统一。
          • 这明显是借口，因为glibc不需要这么做。
      • 关键在哪里？HAL层。
        • google通过HAL层来缓解业界对GPL的恐惧。
        • 在android中，HAL本质上是用户空间的驱动。
        • 目的是规避GPL。
        • 硬件厂商可以把自己的产权相关代码放在HAL层，就可以二进制发布。
        • HAL层细节, 驱动的主要业务逻辑剥离出来放在用户空间的主驱动模块中，内核的驱动是个“影子”驱动，只有透传控制命令和数据的功能。
      • 这条显而易见的路，为什么要等到google来才走得通？
        • 关键还是在bionic libc。
        • google通过bionic暴露更多的接口，包括原来在内核空间才能使用的接口。
        • 其目的是为了让设备驱动能够充分的利用内核和硬件资源。 
      • bionic libc是android的后门.
        • 借这个后门，隔离了GPL，架空了linux.
        • 推动无数硬件厂商为它写驱动，提供器件。

  • APP项目文件夹的组成
    • src：存放项目的源代码。
    • res：存放项目需要的资源文件，比如字符串，布局，皮肤等，这里面每个资源的索引都保存在R.java类中。
      • drawable：存放图片资源，每添加一个图片资源，R.drawble类中对应添加一个图片索引。不同的drawable文件夹存放不同分辨率的图片。
      • layout：存放界面的布局，每一个布局文件对应R.layout里面的布局索引，索引名称与布局文件名称一致。
        • 布局中文件每个控件的id也会被添加到R.id类中去。其中在为控件指定id时有两种方式，一种是@id/btn ，另外一种是@+id/btn ，前一种是R.id类中已经有了这个id定义，后一种是新增一个btn的id索引。
        • Layout目录一般有多个, 适配不同的分辨率, 比如:
          • -mdpi: 中等分辨率. 目前已经很少见了.
          • -hdpi: 高分辨率
          • -xhdpi: 超高分辨率
          • -xxhdpi: 超超高分辨率
          • -480x320: 适配480x320的屏幕
          • -w820dp: w代表平板设备，820dp代表屏幕宽度
          • -v11: 11这样代表在API(11)，即android 3.0后才会用到的.
      • values：里面存放xml资源文件，有如下数据:
        • demens.xml：定义尺寸资源
        • string.xml：定义字符串资源
        • styles.xml：定义样式资源
        • colors.xml：定义颜色资源
        • arrays.xml：定义数组资源
        • attrs.xml：自定义控件时用的较多，自定义控件的属性！
        • theme主题文件，和styles很相似，但是会对整个应用中的Actvitiy或指定Activity起作用，一般是改变窗口外观的！可在Java代码中通过setTheme使用，或者在Androidmanifest.xml中为<application...>添加theme的属性！
      • animator：存放属性动画的XML文件
      • anim：存放补间动画的XML文件
      • raw:  用于存放各种原生资源(音频，视频，一些XML文件等)，我们可以通过openRawResource(int id)来获得资源的二进制流！其实和Assets差不多，不过这里面的资源会在R文件那里生成一个资源id.
    • assets：assets文件夹里面都存放原始的文件格式，不会在R.java中生成一个ID. 在代码中通过AssetManager来进行访问。比如需要在代码中访问加载一个html文件，或者一个txt文档，那么就需要把html文件和txt文档保存到assets文件下。
    • gen：编译时自动生成的，其中最关键的是 R.java 的静态类，它里面包括很多静态类（内部类），每个静态类中的静态成员名称都对应res文件夹中的一个资源名称，保存着该资源的索引，方便在代码中进行获取资源。包括如下:
      • id：保存布局文件中控件id，
      • layout：保存布局文件的索引，
      • string：保存字符串资源的索引。
      • array类: 保存数组的资源索引;
      • drawable：保存图片索引，
    • bin：存放编译后的apk和资源文件。
    • R.java文件：由aapt工具根据App中的资源文件自动生成, 可以理解为资源字典。

    • 编译系统: bp文件的语法
      • Android 7用ninja系统替代make来管理编译。 其配置文件叫 Android.bp。
      • Android的编译处理流程是: 用blueprint和soong来解析bp文件，将其转换为ninja文件。
        • Android还支持 mk文件, 使用 kati 将mk文件转化为 ninja文件.
        • 最终用 ninja文件来管理编译过程.
      • blueprint和soong是go语言写的. 代码在 build 目录下。
      • Android.bp 格式如下:
        • java_library {
          • name: "services", // 模块名
          • dex_preopt: {
            • app_image: true,
            • profile: "art-profile",
          • },
          • srcs: [ // 指定源代码文件
            • "java/**/*.java",
          • ],
          • static_libs: [ // 依赖的静态库
            • "services.core",
            • "services.accessibility",
          • ],
          • libs: [ // 依赖的动态库
            • "android.hidl.manager-V1.0-java",
            • "miuisdk",
            • "miuisystemsdk"
          • ],
        • }
        • cc_library_shared { // 输出文件
          • name: "libandroid_servers",
          • defaults: ["libservices.core-libs"],
          • whole_static_libs: ["libservices.core"],
        • }
